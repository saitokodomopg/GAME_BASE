# CLAUDE.md — 最上位ルール

## セッション開始時

以下の順番で状態を確認してから作業を始めること。

1. `README.md` を読み、このテンプレートの想定ワークフローを把握する
2. `設計/concept.md` が存在するか確認する
   - **存在しない** → 後述の「ヒアリングフェーズ」に入る
   - **存在する** → `設計/architecture.md` も確認し、`作業メモ/` の最新ディレクトリを確認してから作業を始める

## ヒアリングフェーズ（`設計/concept.md` が存在しない場合）

`設計/concept.md` がない＝まだゲームの方向性が固まっていない状態。
**コードを書き始める前に、必ずヒアリングでドキュメントを固めること。**

### アイデアが思い浮かばない場合の対応

ユーザーが「何を作ればいいかわからない」「思いつかない」と言った場合は、
以下の**3ステップで徐々に絞り込む**形で提案する。

**ステップ1 — 大まかなジャンルを選んでもらう（3〜4択）**

例：以下のどれかに興味がありますか？
- アクション（操作して爽快感・達成感を楽しむ）
- パズル（考えて解く喜びを楽しむ）
- 育成・放置（キャラや数値が育つのを楽しむ）
- その他（ランダム性・探索・リズムなど）

**ステップ2 — ジャンル内の具体的なゲーム像を2〜3案提示する**

選ばれたジャンルに合わせて、実現しやすい具体例を提示する。
各案には「どんな楽しさか」と「作る難しさの目安」を添える。

例（アクションを選んだ場合）：
- A. シンプルな横スクロールジャンプ（★☆☆ やさしい）
- B. 弾避けシューティング（★★☆ ふつう）
- C. 物理演算を使ったパズルアクション（★★★ むずかしい）

**ステップ3 — 選んだ案の「楽しいポイント」を一緒に言語化する**

「Aにします」と言われたら「どんな瞬間が一番楽しそうですか？たとえば
ギリギリでジャンプが決まる瞬間？それとも広い世界を探索する感じ？」のように
ゲームの核心を引き出す質問をして、通常のヒアリングフェーズへ接続する。

> **ポイント**：一度に全案を並べず、1ステップごとに回答を受け取ってから次を出す。
> 選択肢は多くて4つまでにする。

---

### ヒアリングの進め方

- 一度に全部聞かず、**1〜2 個ずつ**質問する（一気に多くを聞かない）
- ユーザーの回答を受けて「理解した内容の要約」を示し、認識ずれがないか確認してから次の話題に進む
- 不明確な部分は「〜ということでよいですか？」と具体的に確認する

### 確認が必要な観点（重要度順）

| 優先 | 確認事項 |
|------|----------|
| 高 | どんなゲームか（ジャンル・ゲームプレイの核心） |
| 高 | プレイヤーが「楽しい」と感じる瞬間はどこか |
| 高 | 最初にリリースしたい最小スコープはどこまでか |
| 中 | 画面・操作の基本イメージ（キーボード？タッチ？） |
| 中 | 見た目の雰囲気（ドット絵？シンプル図形？） |
| 低 | 参考にしたいゲームや映像はあるか |

### ヒアリング完了の条件

以下の3点が揃ったら `設計/concept.md` の草案を提示してよい。

1. ゲームの核心（何をして何が楽しいか）が明確
2. 最小スコープが合意されている
3. 主要な操作・画面構成のイメージが共有されている

## ドキュメント固めフェーズ

ヒアリング完了後、実装に入る前に以下の順序でドキュメントを作成・承認する。
**1ファイルずつ草案を示し、承認を得てから次へ進む。**

```
1. 設計/concept.md の草案提示 → 承認
2. 設計/architecture.md の草案提示 → 承認
3. 作業メモ/ の 1_requirements.md 作成 → 承認
4. 作業メモ/ の 2_design.md 作成 → 承認
5. 実装開始
```

承認前にコードを書き始めてはならない。

## architecture.md の決め方

`設計/concept.md` が承認されたら、`設計/architecture.md` 作成前に以下のヒアリングを行う。
concept.md のヒアリングで判明済みの内容は再度聞かない。

### アーキテクチャ決定のためのヒアリング項目（重要度順）

| 優先 | 質問 | 答えが左右する決定 |
|------|------|--------------------|
| 高 | スマホ・タブレットでも遊べるようにしたいか | タッチ対応・レスポンシブ設計の有無 |
| 高 | マルチプレイ（複数人が同時に遊ぶ）要素はあるか | サーバー・WebSocket の要否 |
| 高 | セーブ・ロード、スコアランキングなどの永続データが必要か | localStorage / バックエンドの要否 |
| 中 | 大量のキャラクター・エフェクトが同時に動くか（100体以上など） | Canvas/WebGL vs DOM の選択 |
| 中 | BGM・SE を使いたいか | Web Audio API / ライブラリの要否 |
| 中 | タイルマップ（マップエディタで作ったステージ）を使うか | Tiled 連携・tilemap ライブラリの要否 |
| 低 | JavaScript / TypeScript の経験はあるか | TypeScript 導入可否・補助説明の量 |
| 低 | 使いたいライブラリや技術スタックの希望はあるか | ライブラリ選定の自由度 |

### ヒアリング完了後のアーキテクチャ提案ルール

以下の判断基準でライブラリ選定を行い、**選んだ理由と却下した選択肢も併記して**提案する。

| 条件 | 推奨スタック |
|------|------------|
| シンプルなゲーム・学習目的・マルチプレイなし | Vanilla JS + Canvas API |
| 本格的なゲームエンジン機能が必要（物理・タイルマップ・アニメーション） | Phaser.js |
| 描画パフォーマンス重視・2D グラフィック中心 | PixiJS |
| マルチプレイあり | 上記 + WebSocket（Node.js サーバー or 外部サービス） |
| TypeScript 可・規模が大きい | 上記 + TypeScript + Vite |

提案は以下の形式でまとめる：

```
## 提案アーキテクチャ

**メインスタック**: [選択したスタック]
**理由**: [ヒアリング結果に基づく根拠]
**却下した選択肢**: [代替案と却下理由]

### 構成要素
- フロントエンド: ...
- ビルドツール: ...（必要な場合のみ）
- データ永続化: ...（必要な場合のみ）
- サーバー: ...（必要な場合のみ）
```

## ファイル構成の考え方

このプロジェクトは **設計** と **作業メモ** の2層で管理する。

| | 設計 | 作業メモ |
|---|---|---|
| 場所 | `設計/` | `作業メモ/` |
| 寿命 | プロジェクト全体 | 1作業単位 |
| 書くタイミング | 決定したとき | 作業開始前 |
| 内容 | What・Why | Now・How |
| Claudeへの役割 | 記憶・制約 | 今日の指示書 |

## ディレクトリ構成

```
リポジトリフォルダ/
├── CLAUDE.md                  # 永続：最上位ルール（このファイル）
├── import/                    # 外部から取り込む資料（読み取り専用）
├── 設計/
│   ├── concept.md             # 永続：ゲームコンセプト・仕様概要（方向性を変えない限り変更不可）
│   ├── architecture.md        # 永続：技術スタック・アーキテクチャの決定事項
│   └── known-issues.md        # 永続：あとでなおしたいことのメモ
├── 作業メモ/                  # 作業用：作業単位でディレクトリを作成・完了後も残す
│   └── [YYYYMMDD]-[HHMM]-[タイトル]/   # 例: 20260404-1000-phase1-tilemap
│       ├── 1_requirements.md    # 今回やること・受け入れ条件・タスクリスト
│       └── 2_design.md          # 実装方針・影響範囲（任意：複雑な作業のみ）
└── src/                       # ソースコード
```

## 永続ドキュメントの管理ルール

### いつ作成するか

| ファイル | 作成タイミング |
|----------|--------------|
| `設計/concept.md` | プロジェクト開始時（ゲーム内容が決まったら最初に作成） |
| `設計/architecture.md` | プロジェクト開始時（技術スタックが決まったら作成）または最初の技術決定時 |
| `設計/known-issues.md` | プロジェクト開始時に空テンプレートとして作成 |

### 作成・更新のルール

- **作成時**：1ファイルごとにユーザーの確認・承認を得てから作成する
- **更新時**：`設計/` 以下のファイルは承認なしに変更しない。変更が必要な場合は内容を報告してから承認を得る
- **concept.md の特別ルール**：ゲームの方向性を変える変更は、新規ファイルで議論・合意してから反映する
- **known-issues.md は例外**：後述の運用ルールに従う

### いつ更新するか

- 作業メモ の作業完了時に `設計/` への反映が必要な決定事項があれば報告し、承認後に更新する

## known-issues.md の運用ルール

`設計/known-issues.md` は「あとでなおしたいこと」を気軽にメモする場所。
他の `設計/` ファイルと異なり、**承認なしで追記・更新してよい**。

### 追記するタイミング

- 実装中にバグや「動くけど気になる挙動」を発見したとき
- 作業メモの作業完了時に「今回は直さなかった問題」が残っているとき

### 書き方

テーブルに1行追加する。No は連番。

```
| 3 | ジャンプ中に壁にめり込むことがある | 2026-05-29 | — |
```

### 解決したとき

「なおした日」列に日付を記入する。行は削除しない（記録として残す）。

### 承認ルールの例外について

このファイルだけ承認不要な理由：問題のメモは「今すぐ直す判断」ではなく「忘れないための記録」のため、会話を止めるほどではない。

## 作業メモの内容

作業開始時に `作業メモ/[YYYYMMDD]-[HHMM]-[タイトル]/` を作成し、以下を置く。

**1_requirements.md** — 何をするか（必須）
- 追加・変更する機能の説明
- 受け入れ条件
- やらないこと（スコープ外の明示）
- `- [ ]` / `- [x]` 形式のタスクリスト

**2_design.md** — どう作るか（任意：複雑な作業のみ）
- 実装アプローチ
- 変更するファイル・コンポーネント
- 影響範囲

## 開発ルール

- ヒアリング → ドキュメント承認 → 実装の順序を崩さない
- `設計/` 以下のファイルは承認なしに変更しない
- 作業開始時は作業メモのディレクトリを作成・参照してから始める
- 完了した作業メモのディレクトリはそのまま残す（仕様の記録として）
- 不明確な点があれば実装を止めてユーザーに確認する（推測で進めない）

## 作業メモ作成時

`1_requirements.md` / `2_design.md` は1ファイル作成するごとにユーザーの確認・承認を得てから次のファイルを作成する。

## 作業完了時

1. `1_requirements.md` のタスクリストをすべて `- [x]` に更新する
2. `設計/` への反映が必要な決定事項があれば報告し、承認を得てから更新する
3. 動作確認・品質チェック（型エラー・lint）の結果をユーザーに報告する
